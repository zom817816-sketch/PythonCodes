# 贪心算法（Greedy Algorithm）

## 概述

**贪心算法**是一种在每一步选择中都采取当前状态下**最优**（即最有利）的选择，从而希望最终结果也是全局最优的算法策略。

贪心算法通常用于解决**最优化问题**，其核心特征是：**局部最优 → 全局最优**。

---

## 基本特征

| 特征 | 说明 |
|------|------|
| **贪心选择性质** | 问题的全局最优解可以通过一系列局部最优的选择来达到 |
| **最优子结构** | 问题的最优解包含其子问题的最优解 |

> ⚠️ 贪心算法**并非对所有最优化问题都适用**。适用前需证明问题满足以上两条性质。

## 经典应用与 Python 实现

### 1️⃣ 活动选择问题（Activity Selection）

> **问题**：给定一组活动，每个活动有开始时间和结束时间，选择尽可能多的互不重叠的活动。

```PythonCodes/Algorithms/GreedyAlgorithms/贪心算法.md
def activity_selection(activities):
    """
    activities: list of (start, end)
    返回最多兼容活动的列表
    """
    # 按结束时间升序排序（贪心策略）
    sorted_acts = sorted(activities, key=lambda x: x[1])

    selected = [sorted_acts[0]]
    last_end = sorted_acts[0][1]

    for start, end in sorted_acts[1:]:
        if start >= last_end:          # 不重叠
            selected.append((start, end))
            last_end = end

    return selected


# 示例
acts = [(1, 3), (2, 5), (3, 6), (5, 7), (6, 8)]
print(activity_selection(acts))  # 输出: [(1, 3), (3, 6), (6, 8)]
```

### 2️⃣ 找零问题（Coin Change — 贪心版本）

> **问题**：用最少数量的硬币凑出指定金额（假设硬币面额满足贪心可用性）。

```PythonCodes/Algorithms/GreedyAlgorithms/贪心算法.md
def coin_change_greedy(coins, amount):
    """
    coins: 硬币面额列表（降序排列）
    amount: 需要凑的金额
    返回所需硬币数量
    """
    coins = sorted(coins, reverse=True)
    count = 0

    for coin in coins:
        if amount == 0:
            break
        num = amount // coin           # 尽量多用大面额
        count += num
        amount -= num * coin

    return count if amount == 0 else -1  # -1 表示无法凑出


# 示例
coins = [25, 10, 5, 1]
print(coin_change_greedy(coins, 63))  # 输出: 6（25×2 + 10×1 + 5×0 + 1×3）
```

### 3️⃣ 哈夫曼编码（Huffman Coding）

> **问题**：给定字符及其出现频率，构造一棵**带权路径长度最小**的二叉树（最优前缀码）。

```PythonCodes/Algorithms/GreedyAlgorithms/贪心算法.md
import heapq
from collections import Counter

class HuffmanNode:
    def __init__(self, char, freq):
        self.char = char
        self.freq = freq
        self.left = None
        self.right = None

    def __lt__(self, other):
        return self.freq < other.freq


def huffman_coding(text):
    """构建哈夫曼树并返回字符到编码的映射"""
    freq = Counter(text)
    heap = [HuffmanNode(ch, f) for ch, f in freq.items()]
    heapq.heapify(heap)

    # 贪心：每次合并频率最小的两个节点
    while len(heap) > 1:
        left = heapq.heappop(heap)
        right = heapq.heappop(heap)
        merged = HuffmanNode(None, left.freq + right.freq)
        merged.left = left
        merged.right = right
        heapq.heappush(heap, merged)

    # 生成编码
    codes = {}
    def dfs(node, code):
        if node is None:
            return
        if node.char is not None:
            codes[node.char] = code
        dfs(node.left, code + "0")
        dfs(node.right, code + "1")

    dfs(heap[0], "")
    return codes


# 示例
text = "aabbbccccddddd"
codes = huffman_coding(text)
for ch, code in sorted(codes.items()):
    print(f"'{ch}': {code}")
```

### 4️⃣ 分数背包问题（Fractional Knapsack）

> **问题**：物品可分割，在容量限制下最大化总价值。贪心按 **单位重量价值** 排序即可得最优解。

```PythonCodes/Algorithms/GreedyAlgorithms/贪心算法.md
def fractional_knapsack(items, capacity):
    """
    items: list of (value, weight)
    capacity: 背包容量
    返回最大总价值
    """
    # 按单位重量价值降序排序
    items.sort(key=lambda x: x[0] / x[1], reverse=True)

    total_value = 0.0
    for value, weight in items:
        if capacity >= weight:
            total_value += value
            capacity -= weight
        else:
            total_value += value * (capacity / weight)
            break

    return total_value


# 示例
items = [(60, 10), (100, 20), (120, 30)]
print(fractional_knapsack(items, 50))  # 输出: 240.0
```

## 贪心 vs 动态规划

| 对比维度 | 贪心算法 | 动态规划 |
|----------|----------|----------|
| **决策方式** | 每步只做一次选择，不回溯 | 记录所有子问题解，可回溯 |
| **时间复杂度** | 通常较低（O(n log n) 或 O(n)） | 通常较高（O(n²) 或更多） |
| **适用条件** | 贪心选择性质 + 最优子结构 | 重叠子问题 + 最优子结构 |
| **正确性证明** | 通常需要额外证明 | 递推方程天然保证 |
| **经典问题** | 活动选择、哈夫曼编码 | 0-1 背包、最长公共子序列 |

---

## 实践建议

1. **先证明再使用** — 务必确认贪心策略的正确性（常用交换论证法或数学归纳法）。
2. **排序是灵魂** — 很多贪心算法的关键在于找到一个合适的排序/优先级标准。
3. **注意边界** — 0-1 背包问题不可用贪心，但分数背包可以；找零问题在特殊币值下贪心会失败。
4. **与 DP 对比** — 如果贪心不能保证全局最优，考虑动态规划或回溯。

---

> **总结**：贪心算法简洁高效，但不总是正确的。理解其适用边界，搭配严谨证明，才能在合适的场景中发挥最大威力。
